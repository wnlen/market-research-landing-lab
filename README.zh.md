## 这个项目解决什么问题

大多数市场调研永远不会结束。你不断添加数据、精炼分析，推迟那个真正重要的决策。这个模板强制一个不同的契约：调研的目的是产出决策，而不是推迟决策。

---

## 快速开始

**刚从这个模板派生出新项目？一句话启动：**

```
打开 Claude Code，输入：开始这个项目
```

Claude 进行四组访谈（核心技能 → 渠道现状 → 长期护城河 → 启动约束），把答案写入 `docs/00-research-question.md`。

**有既有文档？** 先把它们放进 `research/raw/inbox/`（头脑风暴稿、愿景文档、计划草案、AI 对话记录，任何你已经写过的内容）。每组访谈前 Claude 会读取这些材料，先说出自己的理解，你来反应、纠正或补充——不需要重新口述已有内容。文档是参考材料，不是答案；最终答案仍由你给出。

无论有无文档，Claude 都会在写入文件前**先展示完整内容等你批准**。

> 等效触发词："启动项目" · "初始化这个项目" · "start this project"
>
> 如果 Claude 没有自动识别，说"用 start-project skill 启动"即可。

## 研究管线

每个派生项目都遵循同一个十文档序列：

```
docs/00   研究问题         ──  先定义要回答什么，再开始调研
docs/01   市场地图         ──  赛道结构、细分层级、主要玩家
docs/02   用户细分         ──  谁有这个问题，谁愿意付费
docs/03   竞品地图         ──  首屏卖点、定价策略、薄弱环节
docs/04   短期现金流选项   ──  可形成可信收费场景的路径
docs/05   长期护城河选项   ──  值得积累的可防御资产
docs/06   融合策略         ──  两条路可以共用哪些资产
docs/07   MVP 候选方案     ──  可横向比较的选项
docs/08   最终决策         ──  冻结的主方向与排除理由
docs/09   执行路线图       ──  调研在这里结束
```

---

## 三段式规则

每个文档里的每条记录都必须分离：

```
证据   ──  你实际观察到的内容。需要来源。
推断   ──  基于证据的有限判断。需要标注置信度。
决策   ──  继续 / 淘汰 / 下一步验证步骤。
```

证据不足必须直接说明。推断不得写进证据段。没有例外。

---

## 毕业条件

当以下六个问题都有具体的、可执行的答案时，停止调研，切换到执行模式：

- [ ] 第一版做什么——具体的产品或服务，不是方向
- [ ] 第一版明确不做什么——写出有可验证理由的排除项
- [ ] 最小演示形态——一个人可独立推进
- [ ] 第一版上线形态——可上线、可承接真实用户、达到收费标准的最小版本
- [ ] 第一个收费点——具体金额和付费场景
- [ ] 短期项目如何反哺长期护城河——具体资产流向，不是口号

## 使用方式

1. 点击 **Use this template** → 创建新仓库
2. 克隆你的新仓库
3. 打开 Claude Code → 输入 **"开始这个项目"**
4. 完成四轮访谈（约 5 分钟）
5. 先读 `docs/README.md`，再核对 `docs/00-research-question.md`
6. 从 `docs/01` 开始填写调研内容

**保持底座仓库干净。** 不要在这里做特定领域的调研。每个项目派生一个独立仓库，一个项目对应一个仓库。

---

## 这个项目不是什么

- 不是产品代码仓库
- 不是写前端、后端或 App 代码的地方
- 不是在没有来源的情况下写市场结论的地方
- 不是没有决策目标的开放式调研工具
- 不是从假设自动生成结论的系统

---

## docs 入口

- `docs/README.md` 是初始化完成后的文档入口。
- `docs/00-09` 是固定的研究到决策主链。
- `docs/product/` 放产品形态和第一版上线设计。
- `docs/audits/` 放仓库与决策链路审计。

---

## 多 Agent 支持

`ai/` 目录包含 Claude Code、Codex 和 ChatGPT 的交接模板。每个 Agent 在同一文档结构上运作，各自的角色职责在 `AGENTS.md` 中定义。
