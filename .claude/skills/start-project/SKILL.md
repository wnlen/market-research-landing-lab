# start-project

## 什么时候触发

用户在派生项目里说下列意思时：

- "开始这个项目" / "启动项目" / "初始化这个项目"
- "start this project" / "let's begin"
- 在克隆/派生的空仓库里首次开口要你动手

## 什么时候**不**触发

- `docs/00-research-question.md` 的"执行人前提确认"已经没有 `[待填写]` 标记 → 项目已初始化过，直接问用户是要重新初始化还是继续推进调研
- 当前仓库就是底座本身（仓库名 = `market-research-landing-lab` 且没有 domain-specific 内容）→ 提醒用户这里是底座，应该先用 Template 派生

## 工作流

### Step 1：前置检查

1. 确认存在 `docs/00-research-question.md`
2. 检查"执行人前提确认"章节是否还有 `[待填写]` 标记。如果没有，停下来问用户是否要重填
3. 读一遍章节结构，记住每个字段的位置

### Step 2：访谈

按顺序问用户以下四组问题。**不要一次丢全部问题**；每组问完、用户答完，**读回一句简短复述**再进入下一组，防止理解漂移。

自由文本字段用普通对话提问；离散选项用 `AskUserQuestion`。

#### 组 1 — 核心技能（自由文本）

> "先定执行人画像。请列出你的核心技能、已有可资产化的成果/作品、相邻但非核心的能力各 1-3 条。写具体，别写'熟悉 AI'这种模糊描述。"

字段对应：
- `核心技能` 列表第 1 条：主要技术/领域能力
- 第 2 条：已有可资产化成果/作品/开源项目/过往交付物
- 第 3 条：相邻但非核心的能力

#### 组 2 — 渠道现状（混合）

先用 `AskUserQuestion` 问三个离散的：

1. 是否有稳定用户池/粉丝/社群/邮件列表？（有 / 无 / 有但规模小）
2. 是否有成熟流量渠道？（有 / 无 / 有但仅限某一平台）
3. 短期现金流能否依赖自然增长？（能 / 不能）

然后自由文本：
- "早期可触达的用户画像是谁？具体到身份标签（如'独立开发者'、'Claude Code 用户'、'跨境电商运营'）"

最后若用户"不能依赖自然增长"，再用 `AskUserQuestion` 问产品形态倾向（可多选）：
- 交付型 / 咨询型 / 模板型 / 工作流型 / 订阅型

#### 组 3 — 长期护城河倾向（自由文本）

> "现在说长期项目。回答三个：
> 1. 这个长期项目叫什么（代号或定位名）
> 2. 一句话描述它是什么、面向谁、解决什么问题
> 3. 12 个月后希望积累的核心护城河资产类型有哪些（例：独家数据、用户关系、专有内容、模型能力、工作流沉淀、品牌声誉、决策记录）"

#### 组 4 — 三项启动约束（离散为主）

用 `AskUserQuestion`：
1. 可用启动时间：全职 / 兼职每周 >20h / 兼职每周 10-20h / 兼职每周 <10h
2. 可用启动资金上限：<5千 / 5千-2万 / 2万-10万 / >10万 / 暂不限制
3. 短期现金流最低月收入目标：<1千 / 1千-5千 / 5千-2万 / >2万 / 暂无硬指标

### Step 3：写入文件

用 `Edit` 工具把答案替换进 `docs/00-research-question.md` 的对应 `[待填写]` 占位。

**严守规则：**
- 用户原话照搬，不要替用户润色或扩展
- 用户没说的字段保留 `[待填写]`，不要自己编
- 待确认项表格里，用户给出答案的项把"待确认"改成具体值；没回答的保持"待确认"

### Step 4：收尾

1. 用 `git diff docs/00-research-question.md` 给用户看一眼变更
2. 问：现在提交推送吗？（可选 yes / 稍后 / 先让我手动调整）
3. 如用户同意提交，commit message 用：
   ```
   chore: 初始化执行人前提，项目 [长期项目代号] 调研启动
   ```

## 边界与约束

- **不要**在访谈里提任何具体方向建议，即使用户犹豫——本 skill 只收集前提，不做方向判断
- **不要**自动跳到下一步调研（填 docs/01 等），除非用户明确说继续
- **不要**在访谈中反复确认每一个字段，一次对话一次复述即可
- 如果用户打断访谈说"先不填这个字段"或"这部分我不确定"，允许跳过，字段保留 `[待填写]` 并在收尾时列出跳过的字段

## 完成标准

- `docs/00-research-question.md` 的"执行人前提确认"章节中，至少"长期护城河倾向"的"方向名称"和"定位"两个字段有具体答案
- 所有未回答字段显式保留 `[待填写]` 或"待确认"
- 用户看过 diff
- 提交状态明确（已提交 / 用户拒绝提交 / 用户推迟）
