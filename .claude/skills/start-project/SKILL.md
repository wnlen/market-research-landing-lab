# start-project

## 什么时候触发

- "开始这个项目" / "启动项目" / "初始化这个项目"
- "start this project" / "let's begin"
- 在克隆/派生的空仓库里首次开口要你动手

## 什么时候**不**触发

- `docs/00-research-question.md` 的"执行人前提确认"已经没有 `[待填写]` 标记 → 询问用户是否要重新初始化
- 当前仓库是底座本身（`market-research-landing-lab`，无领域内容）→ 提醒用户先用 Template 派生

---

## 工作流

### Step 1：前置准备

1. 确认存在 `docs/00-research-question.md`，读章节结构，记住每个字段位置
2. 检查 `research/raw/` 是否有文件：
   - **有文件** → 全部读取，在脑内形成对四个领域的初步理解。不要告诉用户你抽取了什么默认值，只是为后续提问做准备
   - **无文件** → 冷启动，按 Step 2 正常提问

---

### Step 2：四组访谈

每组的结构：
1. **如果读过文档**：先用一句话说出你的读后理解（"从你放进来的材料里，我对这块的理解是……"），然后问一个针对性问题——补充你没读到的、或纠正你理解有偏差的地方
2. **如果没有文档**：直接问下面的标准问题
3. 用户回答后，**复述一句**再进下一组，防止理解漂移

---

#### 组 1 — 核心技能

**标准问题（无文档时用）：**
> "列三件事，写具体：1) 主要技术/领域能力 2) 已有可资产化的成果/作品/交付物 3) 相邻但非核心的能力"

**有文档时：** 说出你对这三块的读后理解，问用户补充或纠正什么。

字段：核心技能第 1-3 条（对应 `docs/00` 的三个占位）

---

#### 组 2 — 渠道现状

**标准问题（无文档时用）：**

先 `AskUserQuestion`（三个离散选项）：
1. 是否有稳定用户池/粉丝/社群/邮件列表？（有 / 无 / 有但规模小）
2. 是否有成熟流量渠道？（有 / 无 / 有但仅限某一平台）
3. 短期现金流能否依赖自然增长？（能 / 不能）

然后自由文本：早期可触达的用户画像（具体到身份标签）

若"不能依赖自然增长"，追问产品形态倾向（`AskUserQuestion` 多选）：
交付型 / 咨询型 / 模板型 / 工作流型 / 订阅型

**有文档时：** 说出你对渠道现状和用户画像的读后理解，针对性追问你读不准或文档里没说的地方。

---

#### 组 3 — 长期护城河倾向

**标准问题（无文档时用）：**
> "回答三个：1) 长期项目叫什么 2) 一句话描述它是什么/面向谁/解决什么 3) 12 个月后希望积累的核心护城河资产（选 2-4 项）"

**有文档时：** 说出你对项目名称、定位、护城河方向的读后理解，问用户确认哪里对、哪里要纠正，以及你读完还没想清楚的地方。

---

#### 组 4 — 三项启动约束

无论有无文档，都用 `AskUserQuestion` 直接问（这类信息文档里通常没有）：

1. 可用启动时间：全职 / 兼职 >20h/周 / 兼职 10-20h/周 / 兼职 <10h/周
2. 可用启动资金上限：<5千 / 5千-2万 / 2万-10万 / >10万 / 暂不限制
3. 短期现金流最低月收入目标：<1千 / 1千-5千 / 5千-2万 / >2万 / 暂无硬指标

---

### Step 3：写入前确认（强制）

四组全部收齐后：

1. 把将要写入的完整内容展示给用户（格式与 `docs/00` 章节结构一致）
2. 明确问："确认写入吗？"——等用户说"写入" / "确认" / "OK" 等明确指令
3. 用户批准后，用 `Edit` 替换 `docs/00-research-question.md` 的占位
4. 写入完成后运行 `git diff docs/00-research-question.md` 让用户核查

**严守规则：**
- 用户原话照搬，不要润色或扩展
- 用户没说的字段保留 `[待填写]`，不要自己编
- **任何情况不得在用户批准前写入文件**

---

### Step 4：收尾

1. 问：现在提交推送吗？（yes / 稍后 / 先手动调整）
2. 若同意提交，commit message：
   ```
   chore: 初始化执行人前提，项目 [长期项目代号] 调研启动
   ```

---

## 边界

- 不做方向建议，只收集前提
- 不自动跳到下一步（docs/01 等），除非用户明确说继续
- 用户说"先跳过这个字段"→ 允许，字段保留 `[待填写]`，收尾时列出跳过项
- 文档是参考材料，不是答案来源；最终答案由用户给出

## 完成标准

- "执行人前提确认"章节至少"长期护城河倾向"的方向名称和定位有具体答案
- 所有未回答字段显式保留 `[待填写]` 或"待确认"
- 用户批准后才写入
- 用户看过 git diff
- 提交状态明确
