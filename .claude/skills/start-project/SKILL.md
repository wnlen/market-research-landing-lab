# start-project

## 什么时候触发

用户在派生项目里说下列意思时：

- "开始这个项目" / "启动项目" / "初始化这个项目"
- "start this project" / "let's begin"
- 在克隆/派生的空仓库里首次开口要你动手

## 什么时候**不**触发

- `docs/00-research-question.md` 的"执行人前提确认"已经没有 `[待填写]` 标记 → 项目已初始化过，直接问用户是要重新初始化还是继续推进调研
- 当前仓库就是底座本身（仓库名 = `market-research-landing-lab` 且没有 domain-specific 内容）→ 提醒用户这里是底座，应该先用 Template 派生

---

## 工作流

### Step 1：前置检查 + 模式判断

1. 确认存在 `docs/00-research-question.md`
2. 检查"执行人前提确认"章节是否还有 `[待填写]` 标记。如果没有，停下来问用户是否要重填
3. 读一遍章节结构，记住每个字段的位置
4. 扫描 `research/raw/` 目录，统计文件数量

**根据第 4 步结果选择模式：**

| 情况 | 模式 |
|------|------|
| `research/raw/` 为空或无文件 | → **访谈模式**（Step 2A） |
| `research/raw/` 有至少一个文件 | → **文档优先模式**（Step 2B） |

告知用户进入哪个模式，说明原因。

---

### Step 2A：访谈模式（无既有文档）

按顺序问用户以下四组问题。**不要一次丢全部问题**；每组问完、用户答完，**读回一句简短复述**再进入下一组。

自由文本字段用普通对话提问；离散选项用 `AskUserQuestion`。

#### 组 1 — 核心技能（自由文本）

> "先定执行人画像。请列出你的核心技能、已有可资产化的成果/作品、相邻但非核心的能力各 1-3 条。写具体，别写'熟悉 AI'这种模糊描述。"

字段对应：
- 第 1 条：主要技术/领域能力
- 第 2 条：已有可资产化成果/作品/开源项目/过往交付物
- 第 3 条：相邻但非核心的能力

#### 组 2 — 渠道现状（混合）

先用 `AskUserQuestion` 问三个离散的：

1. 是否有稳定用户池/粉丝/社群/邮件列表？（有 / 无 / 有但规模小）
2. 是否有成熟流量渠道？（有 / 无 / 有但仅限某一平台）
3. 短期现金流能否依赖自然增长？（能 / 不能）

然后自由文本：
- "早期可触达的用户画像是谁？具体到身份标签。"

若用户选"不能依赖自然增长"，再用 `AskUserQuestion` 问产品形态倾向（可多选）：
- 交付型 / 咨询型 / 模板型 / 工作流型 / 订阅型

#### 组 3 — 长期护城河倾向（自由文本）

> "回答三个：
> 1. 这个长期项目叫什么（代号或定位名）
> 2. 一句话描述它是什么、面向谁、解决什么问题
> 3. 12 个月后希望积累的核心护城河资产类型（2-4 项）"

#### 组 4 — 三项启动约束（离散）

用 `AskUserQuestion`：
1. 可用启动时间：全职 / 兼职 >20h/周 / 兼职 10-20h/周 / 兼职 <10h/周
2. 可用启动资金上限：<5千 / 5千-2万 / 2万-10万 / >10万 / 暂不限制
3. 短期现金流最低月收入目标：<1千 / 1千-5千 / 5千-2万 / >2万 / 暂无硬指标

访谈收齐后 → 进入 **Step 3**。

---

### Step 2B：文档优先模式（有既有文档）

> 用户的角色是**确认、纠偏、裁决**。你的角色是**读文档、压缩信息、形成默认答案**。不要让用户重新口述已有内容。

#### 2B-1. 读取文档

读取 `research/raw/` 目录下所有文件。优先读取文件名含以下关键词的（按优先级）：
- `final` / `最终` / `形态` / `MVP` / `定位` / `规划` / `方案`
- `skill` / `agent` / `backend` / `后端`
- 其余文件按文件名字母序读取

#### 2B-2. 抽取四组默认答案

为每一组输出一张表：

```
### 第 N 组：[组名]

| 字段 | 从文档抽取的默认值 | 来源文件 | 置信度 |
|------|------------------|----------|--------|
| ...  | ...              | ...      | 高/中/低 |

缺失或冲突项：
- [列出文档里没有覆盖、或多个文档说法不一的字段]
```

四组全部输出完后，再集中问**最小确认问题**：

- 只问置信度为"低"或存在冲突的字段
- 每组最多一个问题；能用 `AskUserQuestion` 的用选项而不是开放大问题
- 绝对不要在文档已有明确答案的字段上再问开放问题

#### 2B-3. 等待用户确认

用户对每组的回应只有三种：

1. **"确认"** / 沉默通过 → 按默认值写入
2. **"改第 X 项"** → 记录用户给出的修正值
3. **"我来重写"** → 记录用户自己给的内容

全部四组确认完毕后 → 进入 **Step 3**。

---

### Step 3：写入前确认（必须）

**写入前，必须先展示将要写入的完整内容，等用户明确批准后再操作文件。**

1. 把四组答案整理成"将要写入的内容"展示给用户（格式与 `docs/00` 的章节结构一致）
2. 明确询问："确认写入吗？"——等用户说"写入" / "确认" / "OK" 等明确指令
3. 用户批准后，用 `Edit` 工具替换 `docs/00-research-question.md` 的对应占位
4. 写入完成后，运行 `git diff docs/00-research-question.md` 让用户核查实际变更

**严守规则：**
- 用户原话照搬，不要润色或扩展
- 用户没说的字段保留 `[待填写]`，不要自己编
- 待确认项表格里，有答案的改成具体值；没回答的保持"待确认"
- **任何情况下不得在用户批准前写入文件**

### Step 4：收尾

1. 问：现在提交推送吗？（可选 yes / 稍后 / 先让我手动调整）
2. 如用户同意提交，commit message 用：
   ```
   chore: 初始化执行人前提，项目 [长期项目代号] 调研启动
   ```

---

## 边界与约束

- **不要**在访谈或抽取过程中提任何具体方向建议——本 skill 只收集前提，不做方向判断
- **不要**自动跳到下一步调研（填 docs/01 等），除非用户明确说继续
- **不要**在文档已有明确答案的字段上再问开放问题
- 如果用户打断说"先不填这个字段"，允许跳过，字段保留 `[待填写]`，收尾时列出跳过项

## 完成标准

- `docs/00-research-question.md` 的"执行人前提确认"章节中，至少"长期护城河倾向"的"方向名称"和"定位"两个字段有具体答案
- 所有未回答字段显式保留 `[待填写]` 或"待确认"
- 用户批准后才写入文件
- 用户看过 git diff
- 提交状态明确（已提交 / 用户拒绝提交 / 用户推迟）
